# Offline AI Studio v2.0 —— 系统架构设计报告

> **项目代号**：Offline AI Studio  
> **版本**：v2.0.0  
> **技术栈**：C++20 / Qt6 / Multi-Agent / JSON 计划驱动  
> **设计日期**：2026年7月  

---

## 一、项目定位与概述

Offline AI Studio 是一款面向本地/离线环境的智能助手应用程序。与市面上主流的"大模型直连执行"模式不同，本系统采用**"大模型决策 + C++ 调度层执行"**的分离架构：

- **大模型（LLM）** 不直接操作文件、不直接执行命令，仅负责**生成结构化的 JSON 执行计划**
- **C++ 调度层** 负责解析 JSON 计划，按步骤调度各个 Agent 执行，并管理执行状态
- **UI 层** 实时展示计划生成、步骤执行、输出结果的全过程

这种设计确保了系统的**可控性、可追踪性、可调试性**，同时参考了 Harness Engineering 的执行流设计理念。

---

## 二、设计目标与核心原则

| 设计目标 | 具体措施 |
|---------|---------|
| **可控性** | 大模型仅输出决策指令（JSON），不直接接触操作系统，所有执行由 C++ 层完成 |
| **可追踪性** | 每一步骤的 Agent 调用、参数、输出、状态变更均有日志记录和 UI 展示 |
| **可扩展性** | Agent 通过基类抽象，新增 Agent 只需实现接口即可接入调度系统 |
| **可调试性** | JSON 计划可人工审查、可中途停止、可逐步骤查看输出 |
| **用户体验** | 左侧任务列表实时追踪执行进度，右侧输出面板展示每步的详细日志 |

## 四、分层架构详解

### 4.1 LLM 规划层（大模型决策层）

**核心思想**：大模型只负责"思考"和"规划"，不负责"执行"。

当用户发送一条任务请求时，系统构造一个**规划专用 Prompt**，其中包含：
- 所有可用 Agent 及其工具的详细描述
- JSON 计划格式的严格定义
- 规划规则（分步、顺序、参数映射等）

大模型返回一个 JSON 对象，示例：

```json
{
  "goal": "在 D 盘创建 project 目录并写入 README.md",
  "steps": [
    {
      "step_id": 1,
      "agent": "ComputerAgent",
      "tool": "runCommand",
      "args": { "command": "mkdir D:\\project" },
      "description": "创建项目目录"
    },
    {
      "step_id": 2,
      "agent": "FileAgent",
      "tool": "writeFile",
      "args": {
        "path": "D:\\project\\README.md",
        "content": "# Project\n\nThis is a new project."
      },
      "description": "写入 README 文件"
    }
  ]
}
```

**优势**：
- 大模型输出是可审计的 JSON，人类可以审查和修改
- 即使大模型产生幻觉，C++ 层会检查工具名称和参数有效性
- 执行过程中可随时中断，不会留下不可控的系统状态

---

### 4.2 C++ 调度层（任务调度引擎）

调度层由三个核心组件构成：

#### 4.2.1 Planner（计划解析器）

- **职责**：从 LLM 返回的文本中提取 JSON 块，解析为 `TaskPlan` 数据结构
- **容错设计**：
  - 支持从 markdown 代码块中提取 JSON
  - 支持直接 JSON 响应
  - 解析失败时返回空计划，降级为普通对话模式

#### 4.2.2 TaskScheduler（任务调度器）

- **职责**：遍历 TaskPlan 中的步骤，逐个调用对应 Agent 的工具
- **执行流程**：
  1. 接收 TaskPlan，初始化步骤索引
  2. 按 `step_id` 顺序执行每一步
  3. 每一步：查找拥有该工具的 Agent → 调用 `executeTool()` → 收集结果
  4. 步骤执行期间发射信号：`stepStarted` / `stepOutput` / `stepCompleted` / `stepFailed`
  5. 全部完成后发射 `planCompleted`
- **可中断**：支持 `stopExecution()` 立即停止后续步骤

#### 4.2.3 Orchestrator（总控协调器）

- **职责**：连接 LLM 规划层与 C++ 调度层，是系统的"大脑"
- **两阶段模式**：
  - **规划阶段**：发送 Prompt → 接收 JSON → 调用 Planner 解析
  - **执行阶段**：将 TaskPlan 交给 TaskScheduler → 转发所有信号到 UI
- **信号代理**：将 TaskScheduler 的信号透传给 MainWindow，实现 UI 与调度的解耦

---

### 4.3 Agent 执行层（多智能体协作层）

Agent 是系统的执行单元，每个 Agent 专注于一个领域：

| Agent | 领域 | 工具示例 |
|-------|------|---------|
| **FileAgent** | 文件系统操作 | `readFile`, `writeFile`, `listDir`, `createDir` |
| **ComputerAgent** | 系统命令与环境 | `runCommand`, `listFiles`, `getEnv`, `sysInfo` |
| **（可扩展）** | — | — |

**设计规范**：
- 所有 Agent 继承自 `Agent` 基类，实现纯虚接口
- 每个 Agent 暴露一组 `Tool` 对象，包含名称、描述、参数定义
- `executeTool()` 返回 `QVariantMap`，包含 `success`、`result`、`error` 字段
- Agent 被 TaskScheduler 通过工具名称路由调用（而非直接指定 Agent）

## 六、核心数据结构

### 6.1 TaskStep（步骤）

```cpp
struct TaskStep {
    int         stepId;       // 步骤编号
    QString     agent;        // 建议使用的 Agent 名称
    QString     tool;         // 工具名称
    QVariantMap args;         // 工具参数
    QString     description;  // 步骤描述
    StepStatus  status;       // 状态: Pending / Running / Completed / Failed / Skipped
    QString     output;       // 执行输出
    QString     error;        // 错误信息
    QDateTime   startTime;    // 开始时间
    QDateTime   endTime;      // 结束时间
};
```

### 6.2 TaskPlan（计划）

```cpp
struct TaskPlan {
    QString         planId;    // 唯一标识
    QString         goal;      // 任务目标
    QList<TaskStep> steps;     // 步骤列表
    QDateTime       createdAt; // 创建时间
};
```

## 七、关键技术决策

### 7.1 为什么选择 JSON 作为计划格式？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **JSON（本方案）** | 结构化、可解析、大模型支持好、人类可读 | 需要严格约束输出格式 |
| 正则解析文本 | 灵活 | 不可靠、易出错、难扩展 |
| Function Calling | 标准化 | 不是所有模型都支持 |
| 直接执行 | 简单 | 不可控、不可追踪、不安全 |

**结论**：JSON 是大模型输出结构化数据的最佳通用方案，结合 C++ 层的严格校验，兼顾了灵活性和可控性。

### 7.2 为什么采用信号槽机制？

Qt 的信号槽机制天然适合本系统的需求：
- **解耦**：LLM 回调、调度器、UI 完全解耦
- **线程安全**：Qt 的信号槽支持跨线程自动排队
- **实时更新**：步骤执行日志通过信号实时推送到 UI，无需轮询

### 7.3 为什么 Agent 通过工具名称路由而非 Agent 名称？

```cpp
// 方案A：按 Agent 名称路由（不推荐）
step.agent = "FileAgent";
// 问题：大模型需要知道 Agent 名称，增加认知负担

// 方案B：按工具名称路由（本方案）
step.tool = "writeFile";
// 优点：大模型只需选择"做什么"，系统负责"谁来做"
```

TaskScheduler 自动查找拥有该工具的 Agent，实现**职责分离**。

## 八、扩展路径

| 扩展方向 | 实现方式 |
|---------|---------|
| **新增 Agent** | 继承 `Agent` 基类，注册到 Orchestrator |
| **新增工具** | 在现有 Agent 中新增 `Tool` 并更新 `executeTool()` |
| **支持流式输出** | 在 LlmClient 中接入 SSE 流式解析 |
| **计划重试** | 在 TaskScheduler 中增加失败步骤的重试逻辑 |
| **并行步骤** | 扩展 TaskStep 增加 `dependsOn` 字段，实现 DAG 调度 |
| **持久化日志** | 增加数据库层保存历史任务记录 |

---

## 九、总结

Offline AI Studio v2.0 通过**"大模型决策 + C++ 调度执行"**的分离架构，实现了：

1. **安全性**：LLM 不直接操作系统，所有执行经过 C++ 层校验和路由
2. **透明性**：每一步的执行过程都在 UI 中可视化展示
3. **可控性**：支持中断、支持审查 JSON 计划、支持逐步骤调试
4. **扩展性**：Agent 插件化设计，新增能力无需改动核心调度逻辑

该架构为后续接入更多本地模型（如 Ollama、LM Studio）、更多工具类型（如数据库、浏览器控制）奠定了坚实基础。
