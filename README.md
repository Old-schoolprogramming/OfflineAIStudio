# Offline AI Studio 技术文档

> 一个完全离线运行的本地大模型智能助手应用程序

---

## 一、项目概述

### 1.1 项目背景

随着大语言模型（LLM）技术的快速发展，越来越多的场景需要在本地离线环境中使用AI能力。本项目旨在打造一款**完全离线运行**的智能助手应用，支持调用本地部署的大模型，并通过多种专业Agent（智能代理）来完成复杂的实际任务。

### 1.2 核心功能

- ✅ **完全离线运行**：所有计算和数据处理都在本地完成，无需联网
- ✅ **本地大模型支持**：兼容任何OpenAI格式的本地模型服务（LM Studio、Ollama、vLLM等）
- ✅ **多Agent协作**：内置多种专业Agent，各司其职
- ✅ **总控调度**：Orchestrator智能协调各Agent，实现"思考-行动"循环
- ✅ **简约高级UI**：暗色主题设计，现代化界面风格

### 1.3 技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| C++ | C++20 | 核心开发语言 |
| Qt | 6.11.1 | 跨平台GUI框架 |
| CMake | 4.3+ | 构建系统 |
| MSVC | 2022/2026 | Windows编译器 |

### 2.2 核心模块说明

#### 2.2.1 Orchestrator（总控调度Agent）

**作用**：系统的"大脑"，负责协调整体工作流程

**工作原理**（ReAct模式）：
1. 接收用户问题
2. 把问题 + 所有工具描述发给大模型
3. 大模型决定：直接回答 OR 调用工具
4. 如果调用工具 → 找到对应Agent执行
5. 把工具执行结果再发给大模型
6. 重复步骤3-5，直到任务完成

**类比理解**：就像一个项目经理，接到需求后自己思考，需要时就指挥各个专业部门（Agent）去干活，最后汇总结果给客户。

#### 2.2.2 FileAgent（文件操作Agent）

**作用**：专门处理文件和目录相关的操作

**拥有的工具**：

| 工具名 | 功能 | 参数 |
|--------|------|------|
| readFile | 读取文件内容 | path: 文件路径 |
| writeFile | 写入文件 | path, content, append(可选) |
| listFiles | 列出目录内容 | dir: 目录路径 |
| createDirectory | 创建目录 | path: 目录路径 |
| deleteFile | 删除文件 | path: 文件路径 |
| fileExists | 检查文件存在 | path: 文件路径 |

#### 2.2.3 ComputerAgent（系统命令Agent）

**作用**：执行系统命令和获取系统信息

**拥有的工具**：

| 工具名 | 功能 | 参数 |
|--------|------|------|
| runCommand | 执行系统命令 | command, timeout(可选) |
| getSystemInfo | 获取系统信息 | 无 |
| listProcesses | 查看进程列表 | 无 |
| killProcess | 终止进程 | pid: 进程ID |

## 三、Agent与工具系统详解

### 3.1 什么是Agent？

**Agent（智能代理）** 可以理解为一个"专业员工"：
- 每个Agent有自己的专长领域
- 每个Agent有一套自己的"工具"（能力）
- Agent之间通过总控调度来协作

### 3.2 什么是Tool？

**Tool（工具）** 是Agent可以执行的具体功能：
- 每个工具都有名字、描述、参数
- 大模型通过"读说明书"（工具描述）来学会使用工具
- 就像人用APP一样，看了说明就知道怎么用

### 3.3 工具调用的格式

大模型想调用工具时，会输出特殊格式的文本：

```
<工具名(参数1=值1, 参数2=值2)>
```

例如：
```
<readFile(path="D:/test.txt")>
<runCommand(command="dir", timeout=10)>
```

Orchestrator 会从大模型的回复中解析出这种格式，然后调用对应的工具执行。

## 四、大模型交互原理

### 4.1 Prompt工程

**Prompt（提示词）** 是给大模型的"任务说明书"。我们的系统会精心构造Prompt，告诉大模型：

1. **你是谁**（角色设定）
2. **你有什么工具可用**（工具列表+说明）
3. **工具怎么用**（调用格式）
4. **用户的问题是什么**

### 4.2 系统提示词示例

```
你是一个AI助手，你的角色是：AI Assistant

你可以使用以下Agent和它们的工具：

=== FileAgent ===
描述：文件操作代理，负责文件读写和目录管理
可用工具：
- readFile: 读取文件内容
  参数：
    - path (string) [required]: 文件路径
- writeFile: 写入文件内容
  ...

=== ComputerAgent ===
...

当需要使用工具时，请按以下格式输出：
<工具名(参数名=参数值, ...)>

如果不需要工具，直接回答用户即可。
请始终用中文回答。
```

### 4.3 ReAct思考模式

ReAct = Reasoning（推理） + Acting（行动）

这是当前大模型Agent最主流的工作模式：

```
用户：帮我看看D盘有什么文件
   │
   ▼
思考：用户想知道D盘的文件，我需要调用listFiles工具
   │
   ▼
行动：调用 ComputerAgent.listFiles(dir="D:/")
   │
   ▼
观察：得到文件列表
   │
   ▼
思考：我现在有了文件列表，可以回答用户了
   │
   ▼
回答：D盘有以下文件：...
```

   - 打开应用 → 侧边栏 → 设置

   - 例如：`http://localhost:1234/v1/chat/completions`

## 八、扩展开发指南

### 8.1 如何添加新的Agent？

只需三步：

**第一步**：创建新的Agent类，继承Agent基类
```cpp
class MyAgent : public Agent {
    Q_OBJECT
public:
    QString name() const override { return "MyAgent"; }
    QString description() const override { return "我的新Agent"; }
    AgentType type() const override { return /* 新类型 */; }
    
    QList<Tool*> tools() const override { return m_tools; }
    QVariantMap executeTool(const QString& toolName, const QVariantMap& args) override;
    
private:
    void initializeTools();  // 初始化你的工具
    QList<Tool*> m_tools;
};
```

**第二步**：在initializeTools中添加你的工具
```cpp
void MyAgent::initializeTools() {
    Tool* myTool = new Tool("myTool", "这是一个示例工具");
    myTool->addParameter("param1", "string", "第一个参数");
    m_tools.append(myTool);
}
```

**第三步**：注册到Orchestrator
```cpp
// 在mainwindow.cpp的setupCoreComponents中添加
m_orchestrator->addAgent(new MyAgent(this));
```

### 8.2 设计理念

- **模块化**：每个Agent职责单一，互不依赖
- **可扩展**：新增Agent不需要修改现有代码
- **统一接口**：所有Agent遵循相同的基类接口
- **松耦合**：通过信号槽通信，模块间依赖弱

## 九、常见问题

### Q1：程序启动后没有反应？
A：检查是否配置了正确的本地模型API地址，确保模型服务正在运行。

### Q2：大模型不会调用工具？
A：尝试换一个能力更强的模型，或者调整系统提示词，让工具描述更清晰。

### Q3：可以添加更多Agent吗？
A：完全可以！按照"扩展开发指南"的步骤添加即可，系统设计上支持无限扩展。

### Q4：数据安全吗？
A：完全安全！所有数据都在本地处理，不会上传到任何外部服务器。
